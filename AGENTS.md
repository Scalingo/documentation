# AGENTS.md

## Project Overview

- This repository contains the Scalingo Documentation Center.
- The site is built with Jekyll and served through a Rack/Puma stack.
- Jekyll source files live under `src/` because `_config.yml` sets `source: src`.
- Main content is organized in:
  - `src/_posts` for documentation pages
  - `src/_tutorials` for tutorials
  - `src/_tutorials_categories` for tutorial categories
  - `src/changelog` for changelog entries
  - `src/_includes` for reusable Liquid partials
  - `assets/` for source assets
  - `src/assets/` for built assets

## Deployment Context

- The app is deployed on Scalingo.
- The build uses multiple buildpacks, declared in `.buildpacks`:
  - Node.js buildpack
  - Ruby buildpack
- Node is used to build frontend assets.
- Ruby is used for Jekyll, Rack and Puma.

## Editing Rules

- New markdown files must follow the `YYYY-MM-DD-title.md` naming pattern.
- The date in the filename must be less than or equal to the current build date, otherwise the page may be excluded from the build output.
- When modifying an existing documentation page, update its `modified_at` front matter.
- `modified_at` is typically set to `09:00:00` so the page is positioned correctly in the generated site.
- Do not add `date` in front matter for posts. The filename date is authoritative.
- Do not add `permalink`, `category`, or `categories` in front matter.
- Do not add a first-level heading (`# Title`) in page content. The page title comes from front matter.
- Prefer following the existing content structure and naming conventions already present in nearby files.

## Redirect Rules

- Redirects are defined in the root file `redirections.yml`.
- Any page rename or path change must add a new redirect entry in the `301` section.
- New redirect entries must be inserted above the `obsolete` section.
- Existing redirect entries must never be rewritten, reordered, or removed as part of a rename.
- Treat `redirections.yml` as append-only for normal page moves.
- A missing redirect for a renamed page is a production bug.

Example:

```yml
"301":
  -
    old: "/old/path"
    new: "/new/path"
```

## Build And Validation

- After content or configuration changes, run:
  - `bundle exec jekyll build`
- After asset changes, run:
  - `yarn build`
  - `bundle exec jekyll build`
- When adding a new changelog entry or a new section, prefer a full rebuild instead of assuming incremental regeneration is enough.

## Generated Files

- `src/assets/` contains built assets generated from the sources in `assets/`.
- `src/assets/manifest.json` is generated by the asset build script.
- Do not manually edit generated asset files unless the task explicitly requires it.

## Runtime Notes

- Redirects are loaded at runtime from `redirections.yml` by `config.ru`.
- The Rack layer also handles canonical host logic, SSL enforcement, CORS, and HTML extension rewriting.
- If a change affects URLs or redirects, review both Jekyll output and Rack behavior.

## Working Style For Agents

- Keep changes minimal and consistent with the existing repository style.
- Prefer editing the source markdown or asset source files rather than generated output.
- When changing page locations, update internal links if needed and add the redirect in the same change.
- If a rule in this file conflicts with an explicit user request, follow the user request.
